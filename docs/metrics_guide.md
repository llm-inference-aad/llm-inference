# Metrics Guide

This project reports a few different timing and quality metrics. The important
point is that some are measured in the server, while others are aggregated in
the benchmark runner.

## Latency

### `response_time_sec`
- Measured by the server while generating a response.
- In `server_vllm.py`, this is the generation time returned by vLLM for the
  request.
- In the older HuggingFace server (`server.py`), this is the batch processing
  time for the request batch, not the full end-to-end wall time.

### `e2e_latency_sec`
- Measured by the server from request receipt to response return.
- Includes queue wait, prompt preparation, inference/generation, and response
  packaging.
- This is the best metric for user-visible latency.

### `queue_wait_time_sec`
- Only meaningful when requests are queued in the server.
- Measures time waiting before generation starts.
- For vLLM continuous batching, this is typically near zero in the server.

## Throughput

### `throughput_req_per_sec`
- Computed by the benchmark runner as:

$$
\text{throughput} = \frac{\text{successful requests}}{\text{total benchmark wall time}}
$$

- This is benchmark throughput, not raw model throughput.
- It includes client-side request overhead, server startup if included, and any
  waiting between requests.

## Token Metrics

### `prompt_tokens`
- Number of tokens in the prompt actually sent to the model.

### `completion_tokens`
- Number of tokens generated by the model.

### `total_tokens`
- Sum of prompt and completion tokens.

### `vllm_num_speculative_tokens`
- The speculative token budget requested for a run.
- Useful for comparing speculative vs non-speculative runs.

## Quality / Evaluation

### `evaluationScore`
- A simple format-compliance score from `evaluator.py`.
- Current implementation returns:
  - `1.0` if the output contains a fenced Python code block
  - `0.0` otherwise

This is a format score, not task accuracy.

## Summary Statistics

Benchmark summary files usually report:

- `latency_mean`
- `latency_median`
- `latency_stdev`
- `latency_min`
- `latency_max`
- `latency_p95`
- `latency_p99`

These are computed from the per-request latency samples in the benchmark run.

## Pareto Fronts

For this project, the most useful Pareto front is usually:

- minimize `latency_mean`
- maximize `evaluationScore`

Because `evaluationScore` is often binary and many runs hit `1.0`, the Pareto
front often collapses to the fastest configs among the equally-valid ones.

You can also compare:

- minimize `latency_mean`
- maximize `throughput_req_per_sec`

or use a 3-way tradeoff plot with latency, throughput, and score.

## RAG Data Sources

The centralized RAG vector DB is built from two namespaces:

- code namespace: historical mutation records from LLM-guided evolution
- text namespace: PDFs / docs / benchmark corpus chunks

So yes, the RAG code path is using LLM-guided evolution artifacts as one of its
main sources.

# LLMGE Log Analysis Guide & Debugging Cheatsheet

This document provides a field guide to interpreting the logs generated by the LLM-Guided Evolution (LLMGE) pipeline (`slurm-main-*.out` and `slurm-server-*.out`). It includes real anecdotal examples and troubleshooting steps.

---

## 1. Evolution Loop Logs (`slurm-main-*.out`)

These logs track the "client" side: population management, LLM request submission, and fitness evaluation.

### A. "Generated module failed validation"
**Anecdote:**
```text
********************************************************************************
*                      Generated module failed validation                      *
********************************************************************************
Attempt 1 validation error: name 'nn' is not defined
```

**Diagnosis:**
The LLM generated Python code, but it failed the `validate_module_source()` sandbox check.
*   **Why?** The generated code likely used a standard library alias (e.g., `nn.Conv2d`) without including the necessary import (`import torch.nn as nn`) inside the generated block.
*   **System Action:** It will retry generation (up to `LLM_GENERATION_MAX_RETRIES`). If all attempts fail, it triggers a **Fallback**.
*   **User Action:** If this is frequent, update the **Prompt Template** (`templates/ConstantRules.txt`) to explicitly demand imports: *"Ensure all used modules like torch.nn are imported."*

### B. "Fallback to parent code triggered"
**Anecdote:**
```text
********************************************************************************
*                        Fallback to parent code triggered                     *
********************************************************************************
Reason: name 'nn' is not defined
```

**Diagnosis:**
All LLM retry attempts failed validation. To keep the population size stable, the system discarded the broken LLM code and reverted the gene to be an exact copy of its parent.
*   **Implication:** This individual is valid and loadable, but represents **zero innovation** (mutation failed).
*   **Metric Impact:** High fallback rates = low effective diversity. Ideally, this should be <20%.

### C. "No Placeholder Fitness" (The "Zombie Gene" Log)
**Anecdote:**
```text
Checking for: SOTA_ROOT ./models/network_xXx2YNLao57ZBzgiRXbQzS6oV6m.py
	☠ No Placeholder Fitness (Eval Job Missing) for: xXx2YNLao57ZBzgiRXbQzS6oV6m | Status: subbed file
```

**Diagnosis:**
The evolution manager sees the model file exists on disk, but no evaluation job (Slurm job) is currently tracking it.
*   **Context:** This was previously a "kill condition". Now it is a warning.
*   **Causes:**
    1.  **Transient:** The file was just written, and `check4model2run` hasn't run yet to submit the eval job.
    2.  **Inheritance:** The gene is a fallback/clone and inherited fitness directly (so no job was ever needed).
    3.  **Stuck:** `check4model2run` failed to see the file due to file system latency.
*   **Action:** If this persists for >5 minutes for the same gene, check if `network_xXx.py` actually exists and is readable.

### D. "Job Done" vs. "LLM Job Completed"
**Anecdote:**
```text
********************************************************************************
*                             Python code saved ...                            *
********************************************************************************
Job Done
```
*   **"Job Done"**: Printed by `src/llm_mutation.py`. Means the *script* finished running.
*   **"Direct LLM Job Completed"**: Printed by `run_improved.py`. Means the *thread pool worker* successfully returned from the HTTP request.

## 2. Server Logs (`slurm-server-*.out`)

These logs track the "server" side: the inference engine (DeepSeek/Mixtral) provided by `server.py`.

### A. Successful Batch Processing
**Anecdote:**
```text
Processing batch of 4 requests
Request completed in 209.41s
```
**Diagnosis:**
Healthy server operation.
*   **Time:** If `209s` is close to your `LOCAL_SERVER_TIMEOUT` (e.g., 300s), you are in the danger zone for timeout errors.
*   **Action:** Ensure `LOCAL_SERVER_TIMEOUT > (Batch Processing Time + 60s)`.

### B. CUDA Out Of Memory (500 Error)
**Anecdote:**
```text
RuntimeError: CUDA out of memory. Tried to allocate 2.00 GiB...
```
**Diagnosis:**
The batch size or context length is too large for the GPU.
*   **Fix:**
    1.  Lower `BATCH_SIZE` in `.env`.
    2.  Lower `max_new_tokens` (output length) in the request.

## 3. Pipeline Architecture: Where Checks Occur

```mermaid
graph TD
    A[LLM Inference] -->|Raw Text| B[Code Extraction]
    B -->|Python Candidate| C{Validation Check}
    
    C -- Fail (Syntax/Import Error) --> D[Retry Loop]
    D -- Max Retries --> E[Fallback to Parent]
    C -- Pass --> F[Save network_xXx.py]
    
    E --> F
    
    F -->|File Exists| G{Inheritance Check}
    G -- Can Inherit --> H[Set Fitness Directly (No Job)]
    G -- New Gene --> I[Submit Eval Job (sbatch)]
    
    I --> J[Wait for Results]
    H --> K[Done]
    J --> K
```

**Key Takeaway:** Validation (C) happens *before* the file is saved (F). Inheritance (G) happens *before* the job is submitted (I). Logs must be interpreted based on which stage the gene is currently in.

### 2. **Population Size Variation: 16 → 14 → 11 → 8 → 15**

This is **normal evolutionary algorithm behavior**. Here's the lifecycle:

#### **Generation 0: Initial Population**

```
Created: 8 individuals (start_population_size)
After LLM ops: ~16 individuals (crossover produces 2 children per pair)
```

#### **Invalid Removal**

```
Before: 14 individuals
Removed: 3 with INVALID_FITNESS_MAX (training failures)
After: 11 individuals
```

#### **Selection (NSGA-II)**

```
Before: 11 evaluated individuals
Selected: 8 best (population_size parameter)
After: 8 parents for next generation
```

#### **Generation 1+: Variation**

```
Parents: 8
After mating: 8-16 (depends on crossover success)
After mutation: 14-16
After invalid removal: 11-15
After selection: 8
```

The size fluctuates because:
- **Crossover** can produce 0-2 surviving children per pair
- **Mutation** can fail and revert to parent (increases duplicates)
- **Invalid removal** prunes bad individuals
- **Selection** always picks top `k` (ensures stability)

### 3. **Fitness Inheritance: Not Working (BUG FOUND)**

**NO**, fitness inheritance **never triggered successfully** due to an ancestry tracking bug.

#### **Expected Behavior:**

```
⚠️  Gene xXxkHaNGdgIfUxZd93Cw7X9Ye8s is a fallback clone of parent network
   Inheriting fitness (0.xxxx, xxxxxx) instead of re-evaluating...
```

#### **Actual Behavior:**

```
⚠️  Gene xXxkHaNGdgIfUxZd93Cw7X9Ye8s is a fallback, but parent xXxkHaNGdgIfUxZd93Cw7X9Ye8s not yet evaluated
   Will evaluate (parent and child may be in same generation)
```

#### **Root Cause:**

In `run_improved.py:344`, the ancestry is incorrectly tracking the child as its own parent:

```python
# BUG (old code):
GLOBAL_DATA_ANCESTRY[gene_id] = {'GENES':[gene_id], 'MUTATE_TYPE':["CREATED"]}
#                                          ^^^^^^^^ WRONG! Should be 'network'

# FIX (new code):
GLOBAL_DATA_ANCESTRY[gene_id] = {'GENES':['network'], 'MUTATE_TYPE':["CREATED"]}
#                                          ^^^^^^^^^ CORRECT!
```

The fitness inheritance code tries to look up `parent_genes[0]`, which returns the child's own ID. It then looks for that in `GLOBAL_DATA` and finds the child (which hasn't been evaluated yet), so the condition `parent has valid fitness` fails.

#### **Impact:**

- **2 fallback clones in Gen 0** were unnecessarily re-evaluated (wasted ~4 hours GPU time)
- **2 fallback clones in Gen 1** were unnecessarily re-evaluated (wasted ~4 hours GPU time)
- **Total waste:** ~8 hours of GPU compute
- **Good news:** Despite the bug, only 5 total fallbacks across 8 generations (4.4% rate)

#### **Fix Applied:**

The ancestry tracking has been corrected. Future runs will properly inherit fitness from the seed network.

# 04 - Development Guide & Best Practices

This guide outlines the standard procedures for development, including version control, coding conventions, and debugging.

## 1. Git Feature Branch Workflow

To maintain a clean and manageable codebase, all new work must be done on a feature branch.

1.  **Create a Branch**: Always start from an up-to-date `main` or `new_main` branch. Branch names should follow the `feat/{feature-name}` convention.
    ```bash
    # Get the latest code
    git checkout new_main
    git pull origin new_main

    # Create your feature branch
    git checkout -b feat/my-new-feature
    ```

2.  **Commit Changes**: Make small, atomic commits with descriptive messages that follow the **Conventional Commits** specification.
    - **`feat:`**: A new feature.
    - **`fix:`**: A bug fix.
    - **`docs:`**: Documentation changes.
    - **`refactor:`**: Code changes that neither add a feature nor fix a bug.
    - **`test:`**: Adding or correcting tests.
    - **`chore:`**: Build process or auxiliary tool changes.

    *Example:*
    ```bash
    git commit -m "feat: add user authentication endpoint"
    ```

3.  **Submit a Pull Request (PR)**: Push your branch and open a PR against `new_main`. Provide a clear description of the changes, the motivation, and how they were tested.

## 2. LLM Output Validation

A critical part of the system is hardening it against invalid code generated by the LLM. A multi-layer validation pipeline is used to ensure every individual is a valid, executable module.

- **Layer 1: Code Extraction**: A regex-based function (`clean_code_from_llm`) strips markdown and prose, extracting only the Python code from fenced blocks.
- **Layer 2: Syntax Validation**: The extracted code is checked for syntax errors using `compile()`.
- **Layer 3: Runtime Validation**: The code is executed in a controlled environment using `exec()` to catch `NameError`, `ImportError`, etc.

If validation fails, a retry loop with progressively more specific prompts is initiated. If all retries fail, the system falls back to the parent's code to guarantee a valid individual, ensuring the evolution does not crash.

## 3. Debugging on PACE-ICE

Here are some essential commands for debugging on the cluster.

- **Streaming Logs**: Use `tail -f` to watch log files in real-time.
    ```bash
    # Watch the server log as it starts up
    tail -f slurm-results/slurm-server-*.out

    # Watch the main evolution log
    tail -f slurm-results/slurm-main-*.out
    ```

- **Searching Logs**: Use `grep` to find specific patterns in your log files.
    ```bash
    # Find all syntax errors in the main log
    grep -n "SyntaxError" slurm-results/slurm-main-*.out

    # Count how many times the fallback mechanism was triggered
    grep -c "Fallback to parent" slurm-results/slurm-main-*.out
    ```

- **Checking Job Status**: Use `squeue` to see the status of your jobs.
    ```bash
    # See all your pending and running jobs
    squeue -u $USER
    ```

- **Canceling Jobs**: Use `scancel` to stop a running job.
    ```bash
    # Cancel a specific job by its ID
    scancel 1234567

    # Cancel all your jobs
    scancel -u $USER
    ```
